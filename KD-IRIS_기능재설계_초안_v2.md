## 1. 왜 지금 "실속이 없는가"

현 시스템은 기능마다 **"API 호출 → 결과 나열"** 한 패턴입니다. 실무자가 매일 답을 원하는 세 질문에 모두 못 답합니다:

1. **무엇이 바뀌었나?** — 어제와 다른 회수/처분/허가/공급 이벤트
2. **그게 우리와 무슨 상관인가?** — 우리 품목·성분·경쟁사·공급처에 닿는가
3. **나는 뭘 해야 하나?** — 누가 언제 무엇을 결정/조치해야 하나

재설계는 이 셋을 **백본**으로 세우는 일입니다. API를 더 붙이는 게 본질이 아닙니다.

## 2. 타깃 아키텍처 — 백본 3 + 표면 기능 3

```
[표면]      S1 조회(Search/Product360)   S2 모니터+알람   S3 AI 코파일럿
                       │                       │                  │
                       └───────────┬───────────┴──────────────────┘
[백본]       B1 자사 품목 마스터  ·  B2 스냅샷+diff 엔진  ·  B3 판단·행동 레이어
                       │
[데이터]     공공데이터포털 50+ API (의약품/의약외품/한약/식품) + 식품안전나라
```

### B1. 자사 품목 마스터 — *모든 것의 조인 키*

엑셀 마스터(`품목_마스터_관리_2026-05-28.xlsx`) 분석 결과 **스키마는 풍부하나 실데이터는 골격**(§11.1 상세). 따라서 마스터는 두 부분으로 다뤄야 합니다:

- **A. 사내 입력 컬럼** (xlsx에서 옴): 품목코드·품목명·전문분류·품목분류(외주/주성분)·유효기간·배치사이즈·포장규격 등
- **B. 공공데이터 enrichment 컬럼** (NO 140/35/145 등으로 자동 채움): item_seq · 허가번호 · 주성분(명+코드) · 분류번호/ATC · 제조사 · 재심사예정일 · EDI · 바코드

**부가 마스터**: ① 경쟁사·관심성분 워치리스트 ② 원료/공급처(CMO·DMF) 목록 — 후속.

**매칭 4차원**: 품목 직접 / 동성분 / 제조소(BIZRNO) / 경쟁사·공급처

> Enrichment가 작동해야 동성분 매칭이 살아납니다. 이게 1차 빌드의 핵심 일.

### B2. 일일 스냅샷 + diff — *모니터링 등뼈*
- 야간 배치: 모니터링 API를 **자사 마스터 키로 좁혀** 호출 → snapshot 테이블 저장
- diff: 오늘 vs 어제 → 신규/변경/소멸 추출
- 부수효과: 대용량 API(특허 12만·재심사 11만)도 자사 키 좁힘으로 한도 문제 자연 해소

### B3. 판단·행동 레이어
이벤트마다 `severity` + `자사영향도(직접/동성분/공급처/경쟁사)` + `권장액션` + `담당팀` + `상태(미확인/검토중/완료)`. 가벼운 워크플로 — 멀티유저 인증 비목표, 팀 공용 수준으로 시작.

### 표면 기능 3
- **S1 조회**: 결과는 항상 자사 마스터와 대조되어 "자사 배지 + 신호등"
- **S2 모니터+알람**: diff 피드 + 부서 알람룰 → **Slack** (5절)
- **S3 코파일럿**: 자사맥락+오늘diff+워치리스트를 RAG 컨텍스트로 주입

## 5. 모니터링·알람 설계 — Slack 단일 채널

### 5.1 이벤트 분류 (severity)
- **CRITICAL**: 자사 품목 직접 회수·판매중지·공급중단
- **HIGH**: 동성분 회수, 자사 행정처분, GMP 만료 임박, 안전성서한, 식품 검사부적합
- **LOW**: 신규허가, 임상/특허 동향(정보성)

### 5.2 자사 영향도 4단계
직접 > 동성분 > 공급처/외주 > 경쟁사

### 5.3 알람 룰 (팀 × 이벤트유형 × 자사영향 × 조건 → 채널 · 빈도)

| 팀 | 룰 | 빈도 |
|---|---|---|
| **의약품QA** | 자사 직접 회수/판매중지 (CRITICAL) | 즉시 |
| 의약품QA | 동성분 회수 (HIGH) | 즉시 |
| 의약품QA | 자사 행정처분 / 안전성서한 자사성분 | 즉시 |
| 의약품QA | GMP 자사 만료 D-90/D-30 | 일 1회 |
| 식품QA | 자사 식품 검사부적합/회수 | 즉시 |
| RA | 자사 품목 재심사 D-90/D-30 | 일 1회 |
| RA | 자사 성분 안전성서한 | 즉시 |
| 구매팀 | 자사 원료 공급처 처분/회수/공급부족 | 즉시 |
| 생산관리팀 | 자사 품목 공급중단 등록 | 즉시 |
| 생산관리팀 | 외주 제조사 행정처분/GMP 만료 | 즉시 |
| 영업 | 경쟁사 처분·회수·신규허가 | 주 1회 다이제스트 |

### 5.4 발송 구현 (Slack 로직만, 채널은 추후 주입)

```python
# alerts.py
def post_alert(team: str, severity: str, payload: dict):
    channel = os.getenv(f"SLACK_CHANNEL_{team.upper()}")  # ex: SLACK_CHANNEL_DRUG_QA
    if not channel:
        logger.info("[DRY-RUN] %s/%s — %s", team, severity, payload)
        return
    slack_client.chat_postMessage(channel=channel, blocks=_render_blocks(payload))
```

- **룰 → 채널 매핑은 환경변수**로(`SLACK_CHANNEL_DRUG_QA`, `SLACK_CHANNEL_RA`, …)
- 채널 미설정 시 **dry-run** (콘솔 로그 + DB의 `alerts_sent` 테이블 적재) → 채널 정해지면 환경변수만 채우면 실발송 전환
- 메시지 포맷: severity 색상 헤더 + 자사품목명 + 사유 + 권장액션 + "조치완료" 버튼(in-app deep link)

### 11.3 데이터 모델 (SQLite — 5 테이블)

```sql
-- 자사 품목 마스터
CREATE TABLE master_item (
  item_code        TEXT PRIMARY KEY,        -- 사내 품목코드
  item_name        TEXT NOT NULL,
  item_name_alt    TEXT,                    -- 품목명2
  category         TEXT,                    -- 전문분류 (전문/일반/의약외품/향정)
  sub_category     TEXT,                    -- 품목분류 (외주/주성분/고형제 등)
  shelf_life_mo    INTEGER,                 -- 유효기간(개월)
  pack_spec        TEXT,                    -- 포장규격
  batch_size       INTEGER,
  active           BOOLEAN DEFAULT 1,
  -- enrichment 결과 (NO 140/35/145로 채움)
  item_seq         TEXT,                    -- NO 140의 ITEM_SEQ
  permit_no        TEXT,                    -- 허가번호
  permit_date      TEXT,                    -- 실허가일 (placeholder가 아닌 진짜)
  reexam_date      TEXT,                    -- 재심사 예정일
  entp_name        TEXT,                    -- 허가권자(보통 광동제약)
  atc_code         TEXT,
  edi_code         TEXT,
  bar_code         TEXT,
  source_api       TEXT,                    -- 'NO140' | 'NO35' | 'NO145' | 'unmatched'
  enrich_status    TEXT,                    -- 'matched' | 'review' | 'unmatched' | 'manual'
  enrich_confidence TEXT,                   -- 'high' | 'medium' | 'low'
  enriched_at      TIMESTAMP
);

-- 품목-성분 매핑 (N:M, enrichment 결과)
CREATE TABLE master_ingredient (
  item_code        TEXT,
  ingr_name        TEXT,
  ingr_code        TEXT,
  qty              TEXT,
  unit             TEXT,
  FOREIGN KEY (item_code) REFERENCES master_item(item_code)
);
CREATE INDEX idx_ingr_name ON master_ingredient(ingr_name);

-- API별 일일 스냅샷 (단일 테이블 + api_id로 구분)
CREATE TABLE snapshot (
  snap_date    DATE,
  api_id       TEXT,        -- 'NO539', 'NO90', 'NO564', 'NO547', 'NO132'
  natural_key  TEXT,        -- API별 자연키 (회수: STD_CD+ITEM_SEQ; 처분: ENTP_NAME+처분일)
  payload      JSON,
  PRIMARY KEY (snap_date, api_id, natural_key)
);

-- diff 결과 = 이벤트
CREATE TABLE event (
  event_id        INTEGER PRIMARY KEY AUTOINCREMENT,
  detected_at     TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  api_id          TEXT,
  event_type      TEXT,                   -- 'recall' | 'sanction' | 'safety_letter' | 'gmp_expiry'
  severity        TEXT,                   -- CRITICAL/HIGH/LOW
  impact_level    TEXT,                   -- 'direct' | 'same_ingredient' | 'manufacturer' | 'competitor'
  matched_item_codes  JSON,               -- 매칭된 자사 품목코드 배열
  matched_ingredients JSON,               -- 매칭된 자사 성분명 배열
  raw_payload     JSON,
  status          TEXT DEFAULT 'new'      -- 'new' | 'reviewing' | 'closed'
);

-- 알람 발송 로그 (dry-run 포함)
CREATE TABLE alert_log (
  alert_id     INTEGER PRIMARY KEY AUTOINCREMENT,
  event_id     INTEGER,
  team         TEXT,
  channel      TEXT,                       -- 채널 ID 또는 'DRY_RUN'
  sent_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  payload      JSON,
  FOREIGN KEY (event_id) REFERENCES event(event_id)
);
```

### 11.4 Enrichment 파이프라인

**입력**: 615개 의약품·의약외품 완제품 (`active=1`)
**호출 순서 (품목당)**:

```
1. 품목명 정규화 (공백·괄호·특수문자) → 'norm_name'
2. NO 140 목록 호출: GET /...?item_name=norm_name&entp_name=광동제약
   ├─ ITEM_SEQ 1건 정확매칭 → matched, confidence=high → 단계 5로
   ├─ 후보 다수 → 품목명2/포장규격으로 재필터 → confidence=medium
   └─ 0건 → 단계 3
3. NO 35(한약제제 허가) 호출 (한약 라인 폴백)
   ├─ 매칭 성공 → source_api='NO35', confidence=medium
   └─ 0건 → 단계 4
4. NO 145(의약외품) 호출
   ├─ 매칭 성공 → source_api='NO145'
   └─ 0건 → enrich_status='unmatched' (수동 검토 큐)
5. ITEM_SEQ 확보 시 NO 140 상세 호출 → REEXAM_DATE, MATERIAL_NAME, ATC_CODE
6. NO 140 주성분 호출 → master_ingredient 테이블 적재 (성분명·코드·함량)
```

**호출 한도**: NO 140 개발계정 10,000/일. 615 × 3호출 = 1,845. 여유 충분.

**정규화 규칙**:
- 공백 제거: "광동 쌍화탕" → "광동쌍화탕"
- 괄호·특수문자 제거: "광동쌍화탕(원)" → "광동쌍화탕"
- 한글/숫자만 유지

**Confidence 등급**:
- **HIGH**: 정규화 후 완전일치 + ITEM_SEQ 1건
- **MEDIUM**: 후보 다수 중 품목명2/포장규격으로 1건 좁힘
- **LOW**: 부분일치 다수 → **수동 검토 큐**로

**수동 검토 큐 UI**: enrichment 결과 페이지에서 "확인 필요" 탭 → 후보 1~3개 카드 → QA 담당자가 confirm/reject.
